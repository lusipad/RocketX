# RocketChatX · AI 管家设计（现行）

> 状态：现行 · 2026-07-19
> 取代早期过度设计的 `ai-proactivity-design`（L0–L5 阶梯、简报流等）。
> 两条底线：**自用够克制**、**开源够低门槛**。

---

## 一、管家是什么

好管家的本质不是能力，是**判断力**。

能力是"能帮你做什么"——总结、提醒、排日程。这是工具属性。
判断力是"知道此刻该不该做、做到什么程度、以什么姿态出现"。这是管家和工具的分水岭。

四条性质定义一个合格的管家：

| 性质 | 含义 |
|------|------|
| **懂分寸** | 忙时不打扰，该提醒时在恰当时机说，不刷存在感 |
| **主动但不越界** | 发现冲突会提前说并给方案，但决定权永远在你手里；花钱、发消息等不可逆操作必须确认 |
| **记性好但嘴严** | 记得你的习惯、承诺、偏好，但这些信息只为你服务，随时可查看和删除 |
| **诚实** | 不知道就说不知道，办砸了就承认；偶尔犯错但诚实，远比总是自信但不可靠好用 |

---

## 二、三层架构

管家的所有行为拆成三层，各司其职：

```
感知层 ──→ 判断层 ──→ 表达层
（发生了什么）  （该不该做）  （怎么说/呈现）
```

| 层 | 职责 | 特征 |
|----|------|------|
| **感知层** | 持续监听信号：消息、时间、状态 | 纯事实，与人格无关 |
| **判断层** | 一条线：**"如果我不说，你大概率会漏，且漏了有实际后果"→ 说；否则闭嘴** | 与人格基本无关，是管家的底层逻辑 |
| **表达层** | 用什么语气、说多少、以什么形态呈现 | **人格在这里生效** |

感知和判断是稳定的基础设施；换一个人格只改表达层，不重写逻辑。

## 三、管家性格

### 3.1 不是五选一的人格卡片，是几个可调的轴

| 轴 | 低端 | 高端 | 影响什么 |
|----|------|------|----------|
| **话量** | 一个标记/红点，点开才看 | 主动说一句话 | 咖啡时间摘要长度、提醒形态 |
| **介入深度** | 只标记问题 | 标记 + 给出方案选项 | "出差撞会议"是只标红，还是附调整方案 |
| **语气** | 正式克制 | 轻松随意 | "您有一项冲突" vs "下周三有点挤" |
| **催促感** | 纯信息陈列 | 带轻微紧迫感 | 承诺到期是灰色小字还是醒目提示 |

四个轴的组合已覆盖从"极简效率"到"温和关怀"的跨度。

### 3.2 默认人格

开箱即用的默认值：**克制、预判、不越界**（接近英式管家气质）。

- 话量：偏低（标记为主，必要时才说一句）
- 介入深度：中等（标记问题 + 简短建议，不铺选项）
- 语气：正式但不僵硬
- 催促感：低（信息陈列，不制造焦虑）

### 3.3 实现方式

人格 = **一段 system prompt**（复用已有 `setPersona` 基础设施）+ **轴参数值**。

Persona prompt 结构（供编辑器模板）：

```
角色定位 → 说话方式 → 行为准则 → 明确的"不要" → 场景示例
```

轴参数以结构化配置存储，表达层模板根据参数值选择措辞密度和呈现形式。

## 四、感知：事件源

管家要有判断力，前提是**看得见你的工作全貌**——不只是 IM 消息，还有你的项目管理、日历、代码协作。

### 4.1 五类事件源

| 事件源 | 具体信号 | 拉取方式 |
|--------|---------|----------|
| **IM 消息** | 新消息、@、群活跃度变化 | RC 实时订阅（已有） |
| **时间信号** | 咖啡时间到、待办到期、日历临近 | 客户端定时器 |
| **外部项目系统** | ADO 新工作项分配给我、状态变更、迭代临近截止、评论 | 客户端轮询（5–15 分钟） |
| **代码协作** | PR 请求我 review、我的 PR 被评论、CI 失败 | 客户端轮询（同上） |
| **用户操作** | 右键记待办、主动问管家、标记重要人/群 | 即时事件 |

### 4.2 外部系统轮询机制

**不是后台服务，是客户端开着就拉、关了就不拉。**

- 运行环境：Tauri 桌面端常驻（你不在电脑前时也不需要被提醒）
- 频率：默认 10 分钟一轮，可调
- 认证：复用已有的 ADO PAT / OAuth（`agentEnvironments` 中的连接配置）
- 关注范围：用户已配置的 `adoProjects`（不扫全组织）

拉取的数据**不另存一份**——归一化成事件后进规则引擎，结果写入待办池或直接触发提醒。

### 4.3 外部信号的判断规则

| 信号 | 规则引擎判断 | 输出 |
|------|-------------|------|
| 新高优先级工作项分配给我 | 重要 → 即时 | 主动提醒 |
| 迭代 2 天后发布，我有未完成项 | 漏了有后果 → 提醒 | 咖啡时间 ② 重点标 |
| 我的 PR 被评论 | 中等 → 汇总 | 咖啡时间 ① |
| PR 请求我 review | 中等 → 汇总（超 24h 升级） | 咖啡时间 ① / 超时则主动提醒 |
| 某工作项状态变成"测试中" | 低 → 静默 | 只更新待办池状态 |
| CI 失败（我的提交） | 漏了有后果 → 提醒 | 主动提醒 |

同样的判据：**"漏了有后果" → 说；否则攒到咖啡时间或静默更新。**

## 五、核心机制

管家的三大职责，按优先级排列：

### 5.1 守门——分流（第一职责，感知 + 判断层）

**你打开客户端，重要的自然浮在眼前，其余退后一步。你甚至感觉不到它在工作。**

判定「重要」= 三条简单规则，你掌控，不靠 AI 猜：

- **重要的人**（手动标记）
- **重点的群**（手动标记）
- **被 @**（自动）

| | 位置 | 提醒 |
|---|---|---|
| 重要的 | 一级列表，正常显示 | 正常通知 |
| 非重要的 | 收进二级 | 不提醒、不亮红点，但**不删除**，随时可翻 |

降噪分两档：
- **规则降噪**（零配置）：分流 + 折叠 + 排序，不调模型，人人开箱即用
- **AI 降噪**（配了模型增强）：长会话压成一句摘要等

### 5.2 汇聚——咖啡时间（第二职责，表达层）

**你坐下来时，散落各处的碎片已经拼好端给你。**

本质是一个「时刻」——到你设定的时间点，把信息汇到一处。**不存数据，只切视图。**

- 时间点自定义，默认 2 个：**9:00**（开工规划）+ **18:00**（收工回顾）
- 可改 / 可关 / 可加

汇总回答三个问题：

| # | 问题 | 内容 | 来源 |
|---|------|------|------|
| ① | 我错过了什么 | IM 非重要消息摘要 + 外部系统新动态 | 二级消息 + ADO/PR 事件 |
| ② | 我答应/在等什么 | 我的承诺（到期标重）+ 我在等别人回的（超时标） | 待办池筛选 |
| ③ | 接下来做什么 | 待办排序 + 迭代剩余 + 今天日程 | 待办池 + ADO 迭代 + 日历 |

### 5.3 开口——主动提醒（第三职责，判断 + 表达层）

**只在"你大概率会漏、且漏了有实际后果"时才开口。其余时间沉默是默认状态。**

触发条件（穷举，不随意扩张）：

| 信号 | 例子 | 表达形态（受人格轴影响） |
|------|------|--------------------------|
| 承诺即将到期 | "给张三的方案明天要交" | 标记 / 一句话提醒 |
| 日程冲突 | 出差和重要会议撞了 | 标记 + 可选方案 |
| 等待超时 | 你请别人做的事，三天没回 | 标记 / 建议催一下 |
| 重要且紧急的 @ | 老板在群里 @ 你 | 即时通知（不进咖啡时间） |
| 新高优先级任务 | ADO 派了个 P0 给你 | 即时通知 |
| 迭代临近截止 | 后天发布，你还有 3 项未完成 | 咖啡时间重点 + 可选提前提醒 |
| CI 失败（我的提交） | 构建红了 | 即时通知 |
| Review 请求超时 | 有人等你 review 超过 24h | 咖啡时间提醒 |

**不触发的情况**：群里水了一百条、工作项状态正常流转、非关键人发了非紧急消息——全沉底或静默更新待办池。

### 5.4 ADO 集成：基于默认模板，零自定义

管家直接读 ADO 默认过程模板（Agile / Scrum）的标准字段，**不要求用户做任何 ADO 配置改动**。

**依赖的标准字段：**

| ADO 字段 | 管家用它做什么 |
|----------|--------------|
| `Assigned To` | 筛选"我的"工作项 |
| `State` | 判断进行中 / 卡住 / 完成 |
| `Priority`（1–4） | 1=即时提醒，2=咖啡时间重点，3–4=静默同步 |
| `Iteration Path` + 迭代日期 | 计算"迭代剩余天数 vs 我的未完成量" |
| `Changed Date` | 检测状态变更事件 |
| `Created Date` | 判断"新分配给我" |

**状态（ADO Agile 默认四态）：**

| ADO 状态 | 管家语义 |
|----------|---------|
| 新建 | 待开始 |
| 活动 | 进行中 |
| 已解决 | 待验证（接近完成） |
| 已关闭 | 完成 |

管家关心的：**新建/活动 = 未完成**（算进迭代压力），已解决/已关闭 = 不再跟踪。

**规则引擎如何使用这些字段：**

| 规则 | 条件 | 产出 |
|------|------|------|
| 新任务提醒 | Assigned To = 我 且 Created/Changed 在上次轮询后 且 Priority ≤ 2 | 即时提醒（P1）或咖啡时间（P2） |
| 迭代压力 | 迭代剩余 ≤ 2 天 且 我有未完成项 | 咖啡时间 ③ 重点标记 |
| 卡住检测 | State = 进行中 超过 N 天无变更（N 可配，默认 3） | 咖啡时间 ② 提示 |
| 完成通知 | 我关注的项状态变为完成 | 静默更新待办池 |

**不做的：**
- 不写回 ADO（只读，不改状态/不加评论/不创建工作项）
- 不要求自定义字段、自定义状态、自定义规则
- 不扫描未分配给我的工作项（隐私 + 性能）

> 原则：**ADO 怎么用是团队的事，管家只是一个安静的读者**——它读标准字段、按标准语义理解、发现对你有后果的变化时才开口。

## 六、承诺与工作排布

**承诺不是独立子系统，是待办的一种。**

- 一条**带截止时间 + 关联到某人/某会话**的待办 = 承诺
- 「我在等别人的」= 反向承诺，同样是一条待办
- 录入方式：
  - IM 消息上右键「记为待办」，自动带上对象和时间
  - ADO 工作项分配给我时，自动进入待办池（不需手动录）
- **不做 AI 自动从聊天中识别承诺**（AI 猜容易错，错了增加清理量）
- 排序：按到期日 + 手动优先级，规则排序，**不用 AI 排**

**工作排布 = 待办池的"今日视图"**：咖啡时间的第③组就是它，结合 ADO 迭代剩余量，不另建系统。

## 八、数据与配置

### 存储

- **待办池 → Tauri 侧 SQLite**：待办会持续积累（ADO 同步项、承诺、历史），localStorage 5MB 上限不够；SQLite 支持结构化查询（按到期日、按人、按状态筛选），待办池天然需要
- **AI 设置、人格参数 → localStorage**（量小、读频繁，不值得走 SQLite）
- **不做多设备同步**（当前无此场景）

### 配置

- **AI 配置极简**：填一个 key 即可用；高级用户可换供应商
- **人格配置**：四轴参数 + 可选 persona prompt
- **外部系统**：仅 ADO，复用 `agentEnvironments` 已有的项目绑定 + PAT/OAuth，不做 connector 抽象
- **全本地/客户端**：无后台服务、无 bot——轮询在客户端进程内完成，关了就停

### 管家 UI

- **顶级视图**：与工作台、通讯录同层级，主导航可达
- **咖啡时间到达方式**：到设定时间点推系统通知；如果错过（客户端当时关着），下次打开客户端时自动进入咖啡时间视图
- 管家视图内承载：咖啡时间汇总、待办池、主动提醒历史

## 九、明确不做

| 不做 | 为什么 |
|------|--------|
| 主动性阶梯 L0–L5 | 管家开口的判据是一条线（"漏了有后果"），不需要六级台阶 |
| AI 自动识别承诺 | 猜错的代价（清理）> 猜对的收益（省一次右键） |
| AI 判断消息重要性 | 不可预测、不可解释；三条规则已够 |
| AI 排优先级 | 规则排序可预测，AI 排序是黑箱 |
| 简报流 / Butler Feed | 咖啡时间已覆盖，无需另一套推送机制 |
| 后台服务 / bot 化 | 客户端轮询已够用，不值得为此引入服务端部署 |
| 全组织扫描 | 只关注你配置的项目，不扫全组织（隐私 + 性能） |
| 五套人格卡片切换 | 轴参数组合已覆盖，不需要固定人设模板 |

## 十、一句话

管家的价值不在于它能做多少事，而在于它**看得见你的工作全貌，并且知道什么时候该说、什么时候该闭嘴**——IM、项目、日历、代码协作的信号全部收进来，重要的留在眼前，其余的按你的节奏汇总，偶尔在你要漏事时轻声提醒一句，然后退回沉默。
