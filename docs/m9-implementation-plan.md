# M9 实施计划：LAN P2P 直传与断网降级

日期：2026-07-17

## 目标与边界

M9 只交付蓝图 §5.1：可信局域网发现、身份绑定、文件直传、失败回退，以及 Rocket.Chat
不可达时的 LAN 消息与恢复回灌。IPMSG 协议属于 M10，不在本版本提前实现。

- LAN 能力只在 Tauri 桌面端启用；Web 端继续走 Rocket.Chat，不伪装具备原生 socket。
- 广播中的 `userId` 和公钥只用于发现，不能建立信任。信任必须来自已认证 Rocket.Chat
  会话中交换并固定的设备公钥。
- 文件传输不经过 Rocket.Chat：1 MiB 分片、最多 4 条 TCP 流、断点续传、分片与整文件
  BLAKE3 校验。P2P 失败时回到现有 Rocket.Chat 上传路径。
- 离线消息的作者是唯一回灌者；接收方不向服务器代发，避免重复和身份错误。

## Spike C 结论

2026-07-17 对本地 Rocket.Chat 8.6.1 实例执行
`pnpm spike:rc-offline-contract`，临时房间已清理：

1. 自定义 `_id` 可用，重复提交后历史中仍只有一条；但第二次 REST 请求返回内部错误，
   不能把“重试请求成功”当作幂等确认。回灌失败后必须按 `_id` 回查，已存在即确认完成。
2. 提交早一天的 `ts` 被 `error-message-ts-out-of-sync` 拒绝，不能还原服务器时间戳。
3. 当前实例返回 `Custom fields not enabled`。Rocket.Chat 官方契约要求管理员在
   `管理 → 工作区 → 设置 → 消息` 启用消息自定义字段并配置校验规则。

因此：回灌不提交 `ts`；启用 `customFields` 时写入 `rocketxOriginalTs` 和
`rocketxOffline`。未启用时，参与 LAN 会话的两端都在 IndexedDB 持久化原始时间，并在
Rocket.Chat 回声到达后按稳定 `_id` 合并。设置页必须显示当前能力诊断，不能把本机回退
描述成跨设备时间同步。

## 实施切片与验证

### 1. 契约与持久化

- 定义 LAN v1 消息、身份、文件清单和错误码；所有帧都有版本、长度上限和会话 ID。
- `outbox` 保存稳定 `_id`、房间、作者、原始时间、正文、投递状态和接收端本地副本。
- 回灌按原始时间排序；发送异常后按 `_id` 回查再决定是否重试。

验证：纯函数覆盖排序、重复合并、作者唯一回灌、损坏记录隔离和 customFields 降级。

### 2. 设备身份与发现

- Rust 生成 Ed25519 设备密钥，私钥进入 OS keychain；前端只得到公钥与设备 ID。
- 通过已认证 Rocket.Chat DM/私有频道发送版本化设备公钥控制消息，接收后按
  `server + userId + deviceId` 固定。
- mDNS `_rcx._tcp.local` 为主发现，UDP 组播为兜底；发现结果只有和固定公钥一致才标为可信。
- TCP 握手对随机挑战和双方会话摘要签名；冒充 `userId` 或替换公钥必须失败。

验证：单元测试覆盖签名成功、伪造 userId、替换公钥、重放挑战、超时与畸形广播。

### 3. 文件直传

- 接收方先返回缺失分片；发送方用最多 4 条 TCP 流传输 1 MiB 分片。
- 临时文件和位图按 transfer ID 持久化；断线后重新协商缺失分片。
- 每片与整文件做 BLAKE3；最终校验通过后原子完成并发出前端事件。
- 发送文件时优先可信同网段设备；无可信设备、握手失败、超时或校验失败则调用现有
  Rocket.Chat 上传，界面展示实际路由与速率。

验证：小文件、跨分片、损坏分片、断点续传、同名文件、取消和 RC 回退；另执行 5 GiB
实机基准并记录 P2P/服务端数字。

### 4. 断网消息与回灌

- Rocket.Chat 发送失败且存在可信 LAN 对端时，消息经 LAN 投递并进入作者 outbox；接收方
  本地持久化相同 `_id` 与原始时间。
- Rocket.Chat 恢复后仅作者回灌；两端收到服务端消息时按 `_id` 移除离线标记，保持
  RocketX 原始时间展示。
- 未参与原 LAN 会话的客户端在 `customFields` 关闭时只能看到服务器回灌时间；设置页明确
  显示这一限制。

验证：模拟 RC 不可达、两端 LAN 收发、进程重启、重复恢复和乱序恢复；服务端历史最终只有
一条，作者与接收方显示原始时间。

### 5. 应用能力与交付门禁

- 能力总线注册 `lan.peers()`（`lan:discover`）和 `lan.send()`（`lan:transfer`）。
- 第三方应用只能看到去敏后的 peer 信息，不能读取私钥、挑战材料或任意本机路径。
- 更新 README、CHANGELOG、架构记录和实现说明。

最终门禁：`pnpm typecheck`、纯函数/回归测试、Rust 测试、Web 构建、桌面构建、真实
Rocket.Chat 冒烟、双桌面 LAN 验收、5 GiB 基准、冒充拒绝、拔网线续传、断服聊天与幂等
回灌。全部通过后才合并 `main`。
